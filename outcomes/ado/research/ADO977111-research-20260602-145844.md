### Description

This User Story requires the creation of two new REST API endpoints in the `kp-csc-service` for fetching and updating corrugator setup details.

**What is the ask?**

1. **GET endpoint** — Fetch all setup details for a given `setupNumber` on a specific `corrugatorName`.
2. **PATCH endpoint** — Update the setup with all given setup details for a specific `corrugatorName` and `setupNumber`.

**API Contracts:**

```
GET  /kp-csc-service/v1/corrugators/{corrugatorName}/setups/{setupNumber}
PATCH /kp-csc-service/v1/corrugators/{corrugatorName}/setups/{setupNumber}
```

**Payload Reference Documents:**

- Feature design: `feature-design-corrugator-link.md` (KP-Xmit-KnowledgeBase repo)
- Business logic contracts: `vue-csc-business-logics-in-xml-contracts.md` (KP-Xmit-KnowledgeBase repo)

**Reference Implementation (Groovy Templates in KP-MapJava):**

- `comms/kp-comms/conf/kiwiplan/xmgen/template/bhs.download_setup_details_knife_include.message.script.template` — for API data construction
- `comms/kp-comms/conf/kiwiplan/xmgen/framework/service_access.framework.script.template` — service access framework

**Expected Behavior:**

- The GET endpoint returns a full representation of a corrugator setup identified by `setupNumber`.
- The PATCH endpoint accepts and applies partial or full setup updates.
- Payload structure and field definitions are described in the knowledge base documents referenced above.

### Domain Questions

**Round 1 — What data fields and configuration parameters are included in a corrugator setup record in the CSC module?**

*Answer:* The domain documentation reveals CSC setup records contain rich configuration including:
- **ORDER PARAMETERS (CSC/OP prefix):** Parameters numbered 1–31+, including: NAME, DESCRIPTION, CSC08 Add flag (None/Some/All), Read Port Timeout (secs), Max attempts to get history, Controller queue protection, Prewarn distance, Cancel Prog to change Status, Skip Cancelled Setups Details, Kiwi Master Flag, Pro-Rata on Running Setup, Minutes prior Start.
- **XDATA record:** Setup-level data is stored in XDATA.DA (ISAM) or XDATA table (MySQL), including starch_type in byte 126, and CSDATA user-defined bytes 276–510 for custom fields.
- **Controller parameters (XMIT/E5 Sigma 5):** Scoring, roll station, ramp configuration, single-facer flutes, speed settings, number of concurrent auto SS, Reserve setup_id range.
- **CSC Integration note:** CSC API currently has three existing endpoints — Lineup, Current Run, Feedback — and is still under development. The Setup endpoints (GET/PATCH) requested in this work item extend this integration.

---

**Round 2 — How does the CSC VUE corrugator link send and receive setup configuration data?**

*Answer:* The VLink documentation clarifies the architecture split:
- **Classic environment:** XMGEN pushes Lineup information to VLink via its API; VLink pushes Feedback and Current Run data to the XMGEN API.
- **VUE environment:** VLink *queries* VUE APIs for Lineup information, and pushes Feedback and Current Run data to the VUE APIs.
- BHS DryEnd 4.2.5 is confirmed operational in CSC and PCS VUE environments.
- The BHS download setup details template (`bhs.download_setup_details_knife_include.message.script.template`) is a Groovy-based script template used by XMGEN/VLink for constructing the API data payload — it represents the reference implementation for what should become the new GET `/setups/{setupNumber}` endpoint response body.

---

**Round 3 — What is the architecture of the CSC corrugator link in XMGEN modernization, and how are existing endpoints structured?**

*Answer:* Extremely relevant findings from the NLA (New Link Architecture) documentation:
- **CSC Integration is still under development** — current endpoints cover Lineup, Current Run, Feedback only.
- **NLA API pattern:** Link Central acts as the gateway, calling CSC/PCS service APIs and forwarding results to Link Device. DTOs (e.g., `JobUpdateReqDTO`, `JobCompleteReqDTO`) are used for API response payloads.
- **Existing CSC/PCS-style API paths follow:** `/kp-pcs-service/v3/machines/{machine-id}/lineup-entries`, `/api/v2/machines/{machineNumber}/feedback/update`, `/api/v2/machines/{machineNumber}/lineup`.
- **Classic/VUE parity rule:** Feedback update for classic is the same for both CSC classic and CSC VUE — both use shadow tables (JBSPEC, STEDIE) in manufacturing VUE and classic databases.
- **Key design precedent for this work item:** The same pattern (create new API in `kp-csc-service`, new DTOs, integration with Link Central) should be used for the Setup GET/PATCH endpoints.

### Code Questions

**Round 1 — `kp-csc-service` structure: what controllers and endpoints currently exist?**

*Answer:* The `kp-csc-service` module is at `csc/kp-csc/kp-csc-service/src/com/kiwiplan/csc/rest/` and contains:

**Existing controllers (all under `controller/v1/`):**
- `CorrugatorOrderController.java` — handles `POST /api/v1/corrugator-orders`, `GET /api/v1/corrugator-orders/{orderNumber}/{jobNumber}`, `DELETE /api/v1/corrugator-orders/{orderNumber}/{jobNumber}`, `PATCH /api/v1/corrugator-orders/{orderNumber}/{jobNumber}/quantities`

**Existing exceptions (reusable for new endpoints):**
- `CorrugatorNotFoundException.java` — throws HTTP 404 when corrugator not found by number
- `CorrugatorSetupNotFoundException.java` — throws HTTP 404 when setup not found (already exists!)
- `CscLabelValidationException.java`

**Existing DTOs:**
- `CorrugatorSetupLineupDTO` — fields: `corrugatorNumber`, `programNumber`, `knives` (List<KnifeDTO>)
- `KnifeDTO` — knife-level data
- `ScoreDTO` — scoring data
- `CorrugatorOrderDTO` — full corrugator order DTO
- `CorrugatorOrderQuantityUpdateDTO` — partial update DTO (PATCH pattern)

**API YAML style** (from `corrugator_order_api.yaml`, OpenAPI 3.0.0):
- Uses `/api/v1/` base path inside controller
- OAuth2 security scopes: `readCorrugatorOrders`, `writeCorrugatorOrders`
- Error schema: `ErrorData` with nested `error.code` and `error.logMessage`
- Response codes: 200/201/204/400/403/404/500

**Round 3 — API contract and controller patterns: what does the OpenAPI YAML and CorrugatorOrderController reveal?**

*Answer:* The `corrugator_order_api.yaml` (OpenAPI 3.0.0) in the `doc/` folder establishes the pattern:
- All error responses use the `ErrorData` component with `error.code` (e.g., `"CORRUGATOR_ORDER_NOT_FOUND"`) and `error.logMessage`
- OAuth2 security scopes follow the naming convention: `readCorrugatorOrders` / `writeCorrugatorOrders` → new scopes would be `readCorrugatorSetups` / `writeCorrugatorSetups`
- PATCH pattern is already implemented in `CorrugatorOrderController`: uses a dedicated `CorrugatorOrderQuantityUpdateDTO` request body and returns the updated resource
- New YAML doc should be `doc/corrugator_setup_api.yaml` (follow same format)
- **Important observation:** The work item uses `corrugatorName` (string) as the path variable, while the existing `CorrugatorSetupLineupController` uses `corrugatorNumber` (integer). The new controller must implement lookup-by-name, which requires verifying how corrugator name maps to corrugator ID in the database.
