### Description

**What the issue is:**

KP-Xmit-LinkCentral was identified in a security remediation review as storing sensitive credentials in plain text within the deployed configuration file `kiwiconf/kiwiplan/linkcentral.yaml`. The specific gaps are:

- Database credentials currently injected via environment variables `${KIWI_DB_USER}` and `${KIWI_DB_PASS}` — these are still plain-text values at runtime.
- Keystore password currently provided via `${KIWI_KS_PASS}` — also a plain-text value.
- KiwiPlan comms credentials are hard-coded in plain text: `username: remuser` / `password: secr8`.

KP-Xmit-LinkDevice and KP-Xmit-VLink-2.0 were confirmed as **not** requiring changes.

**Expected behavior:**

Introduce a **setup wizard** that:
1. Prompts customers/technicians for required configuration values.
2. Encrypts sensitive values using `KP-Library-Java-KiwiplanCryptography` (`PublicKeyUtils.encryptString()`).
3. Writes the encrypted values into `linkcentral.yaml`.

At runtime, LinkCentral decrypts the values using `PublicKeyUtils.decryptToString()` before use. The wizard supports both initial setup and later edits.

**Acceptance Criteria Summary:**

1. A LinkCentral setup wizard is implemented.
2. The wizard prompts for: DB host, DB port, DB username, DB password, Comms username, Comms password, Keystore path, Keystore password.
3. Sensitive values are **not** written in plain text to `linkcentral.yaml`.
4. The following are encrypted before storage: DB username, DB password, Comms username, Comms password, Keystore password.
5. `KP-Library-Java-KiwiplanCryptography` is used for encrypt/decrypt.
6. Runtime startup logic is updated to decrypt and use encrypted values.
7. Wizard supports re-running for edits after initial install.
8. Existing LinkCentral startup behavior is preserved.
9. No changes required in LinkDevice or VLink-2.0.
10. Documentation/usage notes are added.
11. Validation confirms LinkCentral starts with all three types of encrypted credentials.
12. Legacy hard-coded credentials (`remuser`/`secr8`) are removed from production config.

### Code Questions

**Q1: Where is `linkcentral.yaml` loaded and how are DB credentials and comms credentials consumed at runtime?**

**Answer:**

- **Config loading:** `LinkCentralApplication.java` uses `SpringApplicationBuilder` with `spring.config.name=linkcentral` and `spring.config.location=classpath:/,file:./kiwiplan/`. This means at runtime the configuration file is loaded from `./kiwiplan/linkcentral.yaml` (the deployed customer config directory).

- **DB credentials:** `JdbiConfiguration.java` (`src/main/java/com/kiwiplan/link/linkcentral/config/jdbi/JdbiConfiguration.java`) uses `@Value("${spring.datasource.username}")` and `@Value("${spring.datasource.password}")` to inject credentials directly into a Tomcat JDBC pool builder. Currently these come from env vars `${KIWI_DB_USER}` / `${KIWI_DB_PASS}` in `linkcentral.yaml`.

- **Comms credentials:** `CommsClientConfig.java` (`src/main/java/com/kiwiplan/link/linkcentral/comms/CommsClientConfig.java`) uses `@ConfigurationProperties(prefix = "kiwiplan.comms")` with `@NotNull` validation. Maps directly to `kiwiplan.comms.username: remuser` and `kiwiplan.comms.password: secr8` in the YAML. `CommsTrimClientProxy.java` receives these via constructor injection.

- **Keystore password:** `key-store-password: ${KIWI_KS_PASS}` in the `server.ssl` section of `linkcentral.yaml`.

- **Three sample YAML files exist:** `linkcentral.yaml-opal-bhs`, `linkcentral.yaml-para`, `linkcentral.yaml-gopfert-stora` — all follow the same pattern.

**Q2: Is `KP-Library-Java-KiwiplanCryptography` already a dependency, and is there any existing use of `PublicKeyUtils` or encryption for configuration values?**

**Answer:**

- **Not present.** There are zero references to `KiwiplanCryptography`, `PublicKeyUtils`, `encryptString`, or `decryptToString` in the codebase. The library is not listed in `pom.xml`.

- Existing `com.kiwiplan.inf` dependencies in `pom.xml` are: `kp-spring-helper`, `kp-spring-security-service-starter`, `kp-spring-security-client-starter`, `kp-okhttp3`, and `kp-measure`.

- The only encryption code present is in `Connection.java` (comms layer) — it uses `javax.crypto.Cipher` / 3DES for the TrimShark protocol's wire-level session key exchange. This is unrelated to configuration secrets.

- **Conclusion:** `KP-Library-Java-KiwiplanCryptography` must be added as a new Maven dependency. Its artifact ID and group need to be verified from the Kiwiplan internal Maven repository.

**Q3: Where in the startup sequence should decryption be injected, and what Spring Boot mechanism is appropriate?**

**Answer:**

- `LinkCentralApplication.java` is a vanilla `@SpringBootApplication` with no custom `EnvironmentPostProcessor`, `ApplicationListener`, or `CommandLineRunner`.

- The `@Value` and `@ConfigurationProperties` bindings for credentials happen during Spring context refresh, **after** the `Environment` is populated. To intercept before binding, a **Spring Boot `EnvironmentPostProcessor`** is the correct approach:
  - Implement `EnvironmentPostProcessor`, register it in `META-INF/spring/org.springframework.boot.env.EnvironmentPostProcessor` (Spring Boot 2.7+) or `META-INF/spring.factories`.
  - In the post-processor, iterate over all property sources, find values wrapped in an `ENC(...)` marker (or similar agreed-upon prefix), call `PublicKeyUtils.decryptToString()` on each, and replace the value in a `MapPropertySource` with higher priority.
  - This ensures decrypted values are available when `@Value` and `@ConfigurationProperties` are resolved.

- The **setup wizard** should be a separate runnable: either a second main class in the same JAR (activated by a command-line flag like `--setup`) or a standalone utility JAR. It should: prompt for each required value, call `PublicKeyUtils.encryptString()` for sensitive fields, and write the resulting `ENC(...)` values back to `linkcentral.yaml`.

- `@PostConstruct` (used in `MachinePollConfigLoader`, `EnvContext`, `DescriptionRegistry`) runs too late for credential decryption since the datasource and comms beans are already initialised before `@PostConstruct`.
