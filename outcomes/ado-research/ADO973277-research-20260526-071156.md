### Description

This User Story is part of the **NLA (New Link Architecture)** initiative, which is migrating Link/SCP modules (Convertor, Strapper, Conveyor) away from direct LinkCentral database access toward proper REST APIs in the MapJava `kp-pcs-service` module.

**The Ask:**
Review and enhance the existing `kp-pcs-service` v1 Order API endpoints to ensure they meet NLA requirements. The following three endpoints already exist and should be **reused or enhanced** — not rewritten:

| Endpoint | Method | Status |
|---|---|---|
| `/kp-pcs-service/v1/orders/{orderNumber}/summary` | GET | Existing — Reuse or Enhance |
| `/kp-pcs-service/v1/orders/{orderNumber}/jobs/{jobNumber}/steps` | GET | Existing — Reuse or Enhance |
| `/kp-pcs-service/v1/orders/{orderNumber}/jobs/{jobNumber}/steps/{stepNumber}` | GET | Existing — Reuse or Enhance |

**Consumers:** Convertor (Martin MPC3), Strapper, and Conveyor (Ducker, MMTC/Minda) Link modules.

**Constraint:** Any payload extensions must be backward-compatible — do not break existing consumers.

### Domain Questions

#### Q1: What is the NLA (New Link Architecture) and how are Link/SCP modules expected to interact with the `kp-pcs-service` REST API?

**Answer:** NLA separates the machine/protocol communication layer (Link Device) from the manufacturing business logic (Link Central / PCS service). Key principles:
- Legacy XMGEN/Fortran code had direct database access. NLA replaces this with REST API calls.
- `kp-pcs-service` (in MapJava) provides the manufacturing data REST APIs (order info, lineup, feedback, etc.).
- Link Central calls `kp-pcs-service` to retrieve job data and expose it to SCP/classic environments.
- Link Device handles the protocol-level message formatting and lifecycle, calling Link Central APIs.
- Where machines can directly consume REST APIs, Link Device acts as the thin communication adapter.
- Authentication: HTTPS + OAuth 2.0 JWT tokens.
- Reference implementation from ADO 649907 (Isowa): Link Central calls `/kp-pcs-service/v3/machines/{machine-id}/lineup-entries`, then Link Device handles Isowa FEM21 protocol. The Order API (v1) works at the order/job level — downstream of the lineup.

#### Q2: What order and job step data fields do SCP protocols typically require for Convertor, Strapper, and Conveyor link types?

**Answer:** The domain documentation does not enumerate specific field lists per protocol for the v1 Order APIs (these are NLA-era APIs, not classic SCP). Key contextual findings:
- Conveyor links (e.g., Ducker — `ducmess.scp`) require two types of information: (1) which machines a job routes to, and (2) job-level details per machine. Order info is downloaded on lineup change.
- SCP scripts are per-customer/per-site and use "defined words" to control which fields are required — indicating field requirements are protocol-specific.
- For NLA: the pattern from ADO 649907 (Isowa) is to populate "commonly used job detail as mandatory fields, other fields optional" — the DTO design principle favors additive/optional field extension.
- The SCP files `mp3mess.scp` (Convertor/Martin MPC3), `strmess.scp` (Strapper), `ducmess.scp`, `mmtcmess.scp` (Conveyor) are the authoritative sources for required fields — these exist in the `KP-MAP` repository.

#### Q3: What is the recommended approach for safely extending existing REST API response payloads in the kp-pcs-service without breaking existing consumers?

**Answer:** The Kiwiplan API design documentation confirms the following rule:
> *"All APIs are versioned. When backwards-incompatible changes are required, a new version is released. Kiwiplan may enhance the protocol with changes that do not break compatibility (for example, adding new attributes to an existing resource) without introducing a new version."*

This means:
- **Adding new optional fields** to the existing v1 Order API responses is safe and does not require a version bump.
- **Removing fields, renaming fields, or changing field types** would be breaking changes and would require a new version.
- New fields added for NLA consumers should be designed as optional/nullable, so that existing consumers (which may ignore them) are not affected.
- The DTO design principle from NLA reference implementations: "commonly used job detail as mandatory fields, other fields optional."

### Code Questions

#### CQ1: Where are the existing v1 Order API controllers, DTOs, and service classes in `kp-pcs-service`?

**Answer:**

| Role | Path |
|---|---|
| Controller | `pcs/kp-pcs-service/src/main/java/com/kiwiplan/pcs/rest/controller/OrderController.java` |
| Controller Test | `pcs/kp-pcs-service/src/test/java/com/kiwiplan/pcs/rest/controller/TestOrderController.java` |
| DTO module | `pcs/kp-pcs-service-api/src/com/kiwiplan/pcs/service/model/` |

**Three target API methods in `OrderController`:**

| API | Method | Returns | Delegates to |
|---|---|---|---|
| `GET /v1/orders/{orderNumber}/summary` | `getOrderSummary(orderNumber)` | `OrderSummaryJsonDTO` | `OrderSummaryMapper.toDTO(order, jobs)` via `manufacturingService` |
| `GET /v1/orders/{orderNumber}/jobs/{jobNumber}/steps` | `getOrderJobSteps(orderNumber, jobNumber)` | `List<OrderJobStepJsonDTO>` | `OrderJobStepMapper.map(lineupService.getOrderJobSteps(...))` |
| `GET /v1/orders/{orderNumber}/jobs/{jobNumber}/steps/{stepNumber}` | `getOrderJobStep(orderNumber, jobNumber, stepNumber)` | `OrderJobStepJsonDTO` | `OrderJobStepMapper.map(lineupService.getOrderJobStep(...))` |

**Current `OrderJobStepJsonDTO` fields:** orderNumber, jobNumber, stepNumber (via OrderJobStepKey), machineNumber, machineName, customerName, boardName, entryWidth, entryLength, quantityIn, extendedOrderNumber, orderDescription, stepProgressStatus, materialsForStep.

**Current `OrderSummaryJsonDTO` fields:** orderId, orderNumber, masterSpecification, customer, numberPerSet, orderDescription, orderedDate, List<JobJsonDTO>.

#### CQ2: What does the `OrderJobStepMapper` and `lineupService.getOrderJobStep()` fetch — what underlying data model fields are available that could be added to the response?

**Answer:** The `OrderJobStepMapper` maps from `kiwiplan.pcs.model.lineups.OrderJobStepDTO` (PCS JINI DTO). Both DTOs have **identical fields**, meaning the JINI DTO has no unreported fields currently:

| Field | Currently in REST API? |
|---|---|
| orderNumber, jobNumber, stepNumber | Yes (via key) |
| machineNumber, machineName | Yes |
| customerName, boardName | Yes |
| entryWidth, entryLength | Yes |
| quantityIn, extendedOrderNumber, orderDescription | Yes |
| stepProgressStatus | Yes |
| materialsForStep | Yes |

**Implication:** To add new NLA-required fields (e.g., runSpeed, targetRunSpeed, startTime, or other fields from the design doc), the team will need to:
1. Extend `OrderJobStepDTO` (in `pcs/kp-pcs/kp-pcs-api/src/kiwiplan/pcs/model/lineups/OrderJobStepDTO.java`) with new fields
2. Update the `lineupService` implementation that populates the JINI DTO
3. Add new fields to `OrderJobStepJsonDTO` (in `pcs/kp-pcs-service-api/`)
4. Update `OrderJobStepMapper` to map the new fields

Key file paths:
- `pcs/kp-pcs/kp-pcs-api/src/kiwiplan/pcs/model/lineups/OrderJobStepDTO.java` — JINI DTO (source of truth for available data)
- `pcs/kp-pcs-service-api/src/com/kiwiplan/pcs/service/model/OrderJobStepJsonDTO.java` — REST DTO
- `pcs/kp-pcs-service-api/src/com/kiwiplan/pcs/service/model/mapper/OrderJobStepMapper.java` — Mapper (one-way, manual)

### Recommended Next Steps

Based on the full research, here is a prioritised, actionable list for the developer picking up this work:

#### 1. Identify the required NLA fields for each endpoint (highest priority)

Consult the SCP files for Convertor, Strapper, and Conveyor to determine exactly which fields each protocol needs from the Order API:
- `mp3mess.scp` (Martin MPC3 / Convertor) — in `KP-MAP` repository
- `strmess.scp` (Strapper) — in `KP-MAP` repository
- `ducmess.scp` / `mmtcmess.scp` (Conveyor/Ducker/MMTC) — in `KP-MAP` repository

Cross-reference with how the Lineup API design doc describes the step object in the Lineup payload (e.g., `runSpeed`, `targetRunSpeed`, `startTime`).

#### 2. Extend the PCS JINI DTO if new fields are needed

If fields are missing from the current `OrderJobStepDTO`, add them here first:
- `pcs/kp-pcs/kp-pcs-api/src/kiwiplan/pcs/model/lineups/OrderJobStepDTO.java`

Then update the `lineupService` implementation that populates this DTO to fetch the new fields from the underlying manufacturing data model.

#### 3. Add new optional fields to the REST DTO

Add the NLA-required fields to:
- `pcs/kp-pcs-service-api/src/com/kiwiplan/pcs/service/model/OrderJobStepJsonDTO.java`

Follow the NLA DTO principle: mandatory fields stay as-is, new fields are optional/nullable to preserve backward compatibility with existing consumers.

#### 4. Update the mapper

Update `pcs/kp-pcs-service-api/src/com/kiwiplan/pcs/service/model/mapper/OrderJobStepMapper.java` to map the new fields from JINI DTO to REST DTO.

#### 5. Update the OpenAPI spec

Update `pcs/kp-pcs-service/doc/step_api.yaml` to document the new fields added to the Step response schema.

#### 6. Review `OrderSummaryJsonDTO` for NLA gaps

Check whether the `/v1/orders/{orderNumber}/summary` response (`OrderSummaryJsonDTO` + `JobJsonDTO`) needs any additional fields for NLA consumers. The `JobJsonDTO` currently includes: id, jobNumber, status, dueTime, shippingSet, orderedQuantity, orderedSets, piecesPerSet, deliveredQuantity, undeliveredQuantity, lastUpdate, earliestTime.

#### 7. Write / update tests

Update or add tests in `TestOrderController.java` and `TestOrderJobStepMapper.java` to cover:
- The new fields being returned correctly in the API responses
- Null/absent behaviour for existing consumers (backward compatibility)

#### 8. Test with NLA consumer (LinkCentral)

Once changes are done, verify the three endpoints return the expected payloads by calling them from the NLA LinkCentral module (or using the simulator) with Convertor, Strapper, and Conveyor protocols.
