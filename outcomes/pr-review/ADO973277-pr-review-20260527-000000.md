# PR Review Report

## Review Summary

**Overall Result:** Needs Changes Before Merge

**Review Confidence:** High

**Summary:**
PR #42600 implements a new NLA v2 REST API (`/api/v2/orders`) that adds three NLA scheduling fields (`startTime`, `runSpeed`, `targetRunSpeed`) to the Order/Job/Step response shape via new versioned controller, DTO, and mapper classes. The PR avoids breaking v1 consumers by creating separate v2 artefacts. The implementation is generally well-structured, with null-safe mapping and reasonable security posture consistent with the existing v1 NLA pattern. However, a critical REST contract bug exists in `OrderControllerV2.getOrderJobSteps()`: it throws HTTP 404 when the service returns an empty list, which is semantically wrong and contradicts the developer's own test expectation. A corresponding test is broken as a result. Additionally, the JINI-layer `OrderJobStepMapper` changes are not covered by any unit tests, and a `serialVersionUID` introduction carries a backward compatibility risk in the JINI deployment context.

## Security and REST Review

### Security

- **Unauthenticated endpoints**: The v2 order endpoints are added to the non-secured matcher in `WebSecurityConfig` and permitted in `PcsAuthenticationPatternConfigurator`. This is consistent with the existing NLA endpoint pattern (17+ other endpoints bypass authentication the same way). The security approach is defensible for machine-to-machine JINI/REST integration on an internal network.
- **IDOR classification**: Aikido flagged an IDOR concern (Thread 338642). The concern is that any caller can retrieve data for any `orderNumber`, `jobNumber`, or `stepNumber` without ownership verification. Given the deployment model (internal enterprise LAN, no user-facing authentication expected for NLA), this is classified as **pre-existing pattern, not introduced risk** — but the wontfix dismissal should be better documented. See F05.
- **No secrets in code/tests/logs**: Confirmed.
- **No SQL injection risk**: Data flows through JINI service calls, not raw SQL.
- **No sensitive data in logs**: No logging statements added.
- **No validation bypass**: Path variable constraints (`@Positive`, `@NotBlank`) are applied via `@Validated` on the controller, enforced through `ConstraintViolationException → 400` in `ControllerExceptionHandler`.

### REST/API

- **Endpoint paths**: `/api/v2/orders/{orderNumber}/jobs/{jobNumber}/steps` and `.../steps/{stepNumber}` — consistent with existing v1 pattern at `/api/v1/orders/{orderNumber}/jobs/{jobNumber}/steps`. Path prefix follows convention.
- **HTTP methods**: GET for both endpoints — correct.
- **Empty collection handling**: **Violation** — empty list should return 200 OK with `[]`, not 404. See F01.
- **Error responses**: 404 for not-found (correct for single step endpoint), 400 for validation failures — correct.
- **Response DTO backward compatibility**: v1 `OrderJobStepJsonDTO` is not modified. New v2 DTO is additive. No breaking change.
- **Date serialization**: `startTime` is serialized as `yyyy-MM-dd'T'HH:mm:ssXXX` via `@JsonFormat`. This pattern does not include milliseconds — sub-second precision is lost. Acceptable if NLA consumers only need second-level precision; should be documented if consumers require higher precision.
- **OpenAPI documentation**: `@Operation`, `@ApiResponses`, and `@Tag` annotations are present and appropriate.
- **Response content-type**: `produces = "application/json"` correctly declared.

## Final Recommendation

**Needs changes before merge**

### Mandatory before merge

1. **Fix F01** — Remove the `null || isEmpty()` throw in `OrderControllerV2.getOrderJobSteps()`. Return the mapped list (empty or not) with 200 OK. Only throw `OrderJobStepNotFoundException` on a caught `PcsServiceException.ERROR_FETCHING_STEP`. This one change also resolves F02.

2. **Fix F02** — Once F01 is resolved, `getOrderJobSteps_returnsEmptyListWhenNoSteps` should pass correctly as-is (standaloneSetup will return 200 OK with the empty list). No additional test change needed if F01 is fixed.

3. **Fix F03** — Add unit tests to `TestOrderJobStepMapper.java` in `kp-pcs-impl` covering the new field population from `lineupEntry` (non-null lineupEntry, null lineupEntry, partial nulls).

### Recommended before merge

4. **Address F04** — Use `serialver` to compute and set the correct JVM-generated `serialVersionUID` for `OrderJobStepDTO`, or explicitly document the rolling-upgrade impact if `1L` is intentional.

### Post-merge suggestions

5. **F05** — Update the Aikido thread dismissal with a clear explanation of the internal-network, machine-to-machine design decision.
6. **F06** — Remove or replace validation annotations on `OrderJobStepJsonDTOV2` with `@Schema` annotations for documentation clarity.
