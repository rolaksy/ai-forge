## Review Summary

**Overall Result:** Pass with Concerns

**Review Confidence:** Medium

**Summary:**
PR 41887 adds three new REST endpoints (`/start`, `/update`, `/complete`) under `api/v1/machines/{machineNumber}/feedback` to allow NLA machines to report feedback via HTTP instead of the XMGEN comms protocol. The implementation is generally solid — the code is clean, uses existing exception/handler infrastructure correctly, mirrors established security bypass patterns, and includes meaningful unit test coverage. Key concerns are: (1) field names in the request DTOs deviate from the work item description without the work item being updated to reflect the final contract, (2) Swagger annotations are missing the 500 response case, (3) one test coverage gap (`completeFeedback` with wrong machine number), and (4) the Aikido finding about business logic in the controller was formally closed but the concern technically still exists.

### Important — Should Fix

#### Finding I-1: Work Item Description and Request DTO Fields Do Not Match

**Evidence:**
- Work item 973278 description specifies: `goodQuantityIn`, `wasteQuantityIn`, `setupCompleteTime` as input fields.
- Implementation uses: `quantityFed`, `quantityProduced`. `setupCompleteTime` is not accepted as an input.
- `FeedbackUpdateOrCompleteRequestDTO.java` and `FeedbackStartRequestDTO.java` confirm these renamed fields.
- The Aikido thread 334178 (closed) references a prior version where `wasteQuantityIn` was incorrectly mapped to `quantityProduced`. The fix renamed the fields, but the work item was not updated.

**Why it matters:**
- Anyone reading the work item to understand the API contract (QA, other developers, integrators) will see a different field set than what actually exists. This creates downstream confusion and risks integration errors.

**Suggested fix:**
- Update work item 973278 description to reflect: `quantityFed` (was `goodQuantityIn`), `quantityProduced` (was `wasteQuantityIn`), and explicitly note that `setupCompleteTime` is not an input field (handled internally via macop).

---

#### Finding I-2: Missing Test — `completeFeedback` with Wrong Machine Number

**Evidence:**
- `testUpdateFeedback_WrongMachineNumber_Returns404` exists and validates that update returns 404 when the lineup entry is assigned to a different machine.
- `testStartFeedback_WrongMachineNumber_Returns404` exists for the start endpoint.
- No equivalent `testCompleteFeedback_WrongMachineNumber_Returns404` test exists.
- The `completeFeedback` method calls `findLineupEntry()`, which contains the same machine validation path.

**Why it matters:**
- The machine number validation in `findLineupEntry` is exercised for update and start but not for complete. If the validation logic in `findLineupEntry` changes, no test would catch regressions in the complete path.

**Suggested fix:**
- Add `testCompleteFeedback_WrongMachineNumber_Returns404` following the same pattern as `testUpdateFeedback_WrongMachineNumber_Returns404`.

---

#### Finding I-3: Missing HTTP 500 Response in Swagger Annotations

**Evidence:**
- All three endpoints wrap `PcsServiceException`, `XmgenPcsServiceException`, and `NoSuchObjectException` in `PcsServiceDelegateException`.
- `ControllerExceptionHandler.handlePcsServiceException` maps `PcsServiceDelegateException` → HTTP 500.
- The `@ApiResponses` on all three methods only declares `200`, `400`, `404`. No `500` response is documented.

**Why it matters:**
- NLA machine clients won't know to handle a 500 scenario (e.g., JINI service down). Missing documentation for failure modes can cause silent mishandling of errors.

**Suggested fix:**
- Add `@ApiResponse(responseCode = "500", description = "Internal service error (PCS JINI service failure)")` to the `@ApiResponses` on all three endpoint methods.

---

### Suggestions — Nice to Have

- **Swagger `consumes`**: The `@PostMapping` and `@PutMapping` annotations include `produces = APPLICATION_JSON_VALUE` but no `consumes`. Adding `consumes = APPLICATION_JSON_VALUE` would make the API contract explicit, consistent with REST best practices.

- **Consolidate machine+null check in `startFeedback`**: The start method uses `lineupEntry.getPcsMachine() == null || lineupEntry.getPcsMachine().getMachineNumber() != machineNumber` in one condition. The `findLineupEntry` helper (used by update/complete) splits this into two separate `if` blocks. While both are correct, aligning the style would improve maintainability.

- **Aikido SR finding (thread 332286, closed)**: Business validation helpers (`validateMachine`, `findLineupEntry`, `getStartTimeFromFeedback`) are private methods in the controller. The Aikido comment was closed without code change. Architecturally these belong in a service layer. For this PR it is acceptable, but consider moving this logic when the service is next refactored.
