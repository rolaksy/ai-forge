## Findings

| # | Severity | Area | Finding | Evidence | Impact | Suggested Fix |
|---|---|---|---|---|---|---|
| F01 | **Important** | REST Design — Behavioral Contract | `OrderControllerV2.getOrderJobSteps()` returns HTTP 404 when the service returns an empty list or null. This is now confirmed intentional (tests updated to match). REST convention treats an empty collection as a valid 200 OK response, not a not-found resource. Additionally, v1 `OrderController.getOrderJobSteps()` returns 200 OK with an empty list — v2 silently changes this contract. | `OrderControllerV2.java`: `if(orderJobStepDto == null \|\| orderJobStepDto.isEmpty()) { throw new OrderJobStepNotFoundException(...); }`. v1 `OrderController.java` line 135: `return OrderJobStepMapper.map(lineupService.getOrderJobSteps(...))` — no empty-list check. New tests `returns404WhenServiceReturnsEmptyList`, `returns404WhenServiceReturnsNull` confirm intent. | NLA consumers cannot distinguish between "job doesn't exist" and "job exists but has no steps" — both return 404. If a job can legitimately have zero steps (e.g., a newly created order), callers will incorrectly infer the job does not exist. This may cause silent data issues in NLA integration. | Either: (a) align v2 with v1 behavior (remove the empty-list throw, return 200 OK with `[]`), or (b) explicitly document in the OpenAPI spec (`@ApiResponse` and description) that 404 is returned for both "not found" and "no steps" scenarios so NLA consumers can handle this correctly. |
| F03 | **Important** | Test Coverage — JINI Layer | `OrderJobStepMapper.java` (in `kp-pcs-impl`) was modified to populate `startTime`, `runSpeed`, and `targetRunSpeed` from `lineupEntry`. The existing `TestOrderJobStepMapper.java` was not updated and has zero coverage for these three new fields. | `grep startTime\|runSpeed\|targetRunSpeed TestOrderJobStepMapper.java` → no output. `OrderJobStepMapper.java` diff: three null-guarded blocks extracting from `lineupEntry.getPlannedStartTime()`, `getPlannedMachineRunSpeed().getSpeedPerHour().getGrainAmount()`, `getTargetRunSpeed().getSpeedPerHour().getGrainAmount()`. | No regression safety for the critical data extraction path. If `getSpeedPerHour()` or `getGrainAmount()` can return null, a `NullPointerException` would occur in production with no test to catch it. | Add tests to `TestOrderJobStepMapper.java` covering: (a) non-null `lineupEntry` → all three fields populated correctly; (b) null `lineupEntry` → all three fields at default values; (c) `lineupEntry.getPlannedMachineRunSpeed()` is null → `runSpeed` remains 0; (d) `lineupEntry.getTargetRunSpeed()` is null → `targetRunSpeed` remains 0. |
| F04 | **Important** | Backward Compatibility — Serialization | `OrderJobStepDTO` gains `serialVersionUID = 1L`. The class previously had no explicit `serialVersionUID`; the JVM computed a structural hash as the implicit ID. Setting it to `1L` changes the serialization identity. Any JINI node that previously serialized an instance of this class with the JVM-computed ID will fail with `InvalidClassException` when deserializing after upgrade. | `OrderJobStepDTO.java` diff: `+ private static final long serialVersionUID = 1L;`. Class is `implements Serializable` and used in JINI remote calls. | Breaking change during rolling upgrades if client and server run different versions of the class simultaneously. Low risk for stop-the-world deployments; moderate risk in environments with staged upgrades. | Use `serialver kiwiplan.pcs.model.lineups.OrderJobStepDTO` (compiled against the pre-PR version) to compute the actual JVM-generated value and set that instead of `1L`. This preserves backward compatibility. If the deployment model guarantees no mixed-version nodes, document this explicitly. |
| F05 | **Suggestion** | Security — IDOR Dismissal | Aikido Thread 338642 (status: `wontfix`) dismissed with "These are actually for authentication." The reply is technically misleading — the configuration bypasses authentication. The security pattern is consistent with existing unauthenticated NLA endpoints; the dismissal just lacks a clear rationale. | Thread 338642 still open with vague reply. Security pattern confirmed consistent with `/api/v1/plants/**`, `/api/v1/health/**`. | May raise concern in future security audits or Aikido re-scans. Not an immediate runtime risk. | Update reply to: "This is an internal machine-to-machine NLA integration endpoint on a private network. Authentication bypass follows the same established pattern as `/api/v1/plants/**` and `/api/v1/health/**`." |
| F06 | **Suggestion** | Code Quality — Response DTO | Bean validation annotations (`@NotNull`, `@NotBlank`, `@Positive`, `@PositiveOrZero`) on `OrderJobStepJsonDTOV2` have no runtime effect — they are only applied on inbound request bodies, not on outbound responses. | `OrderJobStepJsonDTOV2.java`: `@NotBlank private String orderNumber;`, `@NotNull private ProgressStatus stepProgressStatus;`, etc. Controller uses this class as a return type. | Maintenance confusion for future developers who may assume outbound validation is active. | Replace with `@Schema(requiredMode = Schema.RequiredMode.REQUIRED)` for documentation intent, or remove the annotations. |

**Severity key:** Critical = must fix before merge · Important = should fix · Suggestion = nice to have
