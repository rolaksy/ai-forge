# Documentation Instructions

## General Rules

- Update documentation when adding or changing features, workflows, scripts, configuration, or notable behavior.
- Keep documentation concise, practical, and easy to follow.
- Use clear markdown headings.
- Use fenced code blocks with the correct language.
- Label multiple code snippets with file path and purpose.

## Diagrams

- Use Mermaid diagrams for architecture.
- Use SVG for non-architecture diagrams.

## Content Expectations

Documentation should include, where relevant:

- purpose
- prerequisites
- setup steps
- usage
- examples
- troubleshooting
- validation or test commands
- known limitations

## Path Rule

Do not hardcode a documentation output path unless the repository already has a clear docs location or the user confirms the target path.

If creating general research documentation, confirm whether to use:

```text

```

or another repository-specific path.
